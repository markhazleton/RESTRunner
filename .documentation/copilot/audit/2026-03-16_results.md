### Security Checklist

- [x] No hardcoded real credentials (detected passwords are sample/seed data only)
- [x] No SQL injection vulnerabilities (file-based storage, no SQL)
- [x] No XSS vulnerabilities detected
- [x] Dependencies free of known vulnerabilities
- [ ] Hardcoded placeholder passwords should be replaced with named constants to avoid accidental promotion
- [ ] `DebugEdit` action in ConfigurationController returns internal model state without authorization check

### Detailed Security Issues

**SEC1–SEC5** — Hardcoded passwords in production/sample data code (not test files).
While these passwords are clearly sample data (values like `"password"` and `"admin123"`) rather than real credentials, storing literal password strings in source establishes a poor pattern. If secret scanning tools (like GitHub secret scanning or truffleHog) are added to the repository, these will generate false positives and noise. The recommended fix is to define a `DomainConstants.SamplePassword` constant or use a clearly named placeholder like `"[sample-password]"`.

**QUAL6** — `DebugEdit` action returns serialized configuration data including users (which contain Password fields) without any `[Authorize]` attribute or route restriction. This could unintentionally expose sample credentials if the application is deployed.
