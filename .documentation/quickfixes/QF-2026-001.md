## Problem Statement

RESTRunner.Services.HttpClient is a custom HTTP request execution service that duplicates functionality now available in WebSpark.HttpClientUtility v2.5.0. The WebSpark package provides production-ready batch execution orchestration, resilience patterns, structured logging, and OpenTelemetry—matching RESTRunner's architecture perfectly. Migrating to WebSpark enables removal of the entire custom service project, reducing maintenance and gaining enterprise-grade features.

## Proposed Solution

Migrate RESTRunner to use WebSpark.HttpClientUtility v2.5.0 `IBatchExecutionService`. Map CompareRunner/CompareInstance/CompareRequest/CompareUser models to WebSpark's batch execution model, update DI registration in console and web apps, migrate ExecutionStatistics to WebSpark's built-in result model, and remove RESTRunner.Services.HttpClient project from solution. Target: complete migration with all tests passing.

## Key Questions for Evaluation

1. **Can WebSpark's `BatchExecutionConfiguration` model directly replace `CompareRunner`?**  
   - Current model: `CompareRunner` → `Instances`, `Users`, `Requests`, iterations → `CompareResult` list
   - WebSpark model: `BatchExecutionConfiguration` → `Environments`, `Users`, `Requests` → `BatchExecutionResult`

2. **Does WebSpark support custom HTTP decorators?**  
   - We currently use headers, correlation IDs, authentication. WebSpark has built-in auth providers and correlation IDs.

3. **Template syntax compatibility:**  
   - Current: `{{property_key}}` for user property substitution
   - WebSpark: `{userId}` syntax (different notation)
   - Can our template model be translated to WebSpark's format?

4. **Statistics aggregation:**  
   - We calculate P50, P75, P90, P95, P99, P99.9 percentiles by hand
   - Does WebSpark provide equivalent percentile calculations?
