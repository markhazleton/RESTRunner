## Problem Statement

RESTRunner.Services.HttpClient was an extra project boundary around request execution that had become costly to maintain. The repository needed to remove that project cleanly while preserving existing CompareRunner execution behavior, statistics aggregation, and console/web integration points.

## Proposed Solution

Evaluate whether WebSpark.HttpClientUtility v2.5.0 batch execution could replace the custom project directly. If that proved incompatible, consolidate the execution service into RESTRunner.Domain, preserve the existing behavior behind `IExecuteRunner`, remove the extra project from the solution, and verify the build and test suite still passed.

## Evaluation Outcome

- WebSpark.HttpClientUtility v2.5.0 was evaluated as the original migration target.
- The package did not expose a usable batch execution API that could replace the existing CompareRunner execution path as initially planned.
- The implementation pivoted to a simpler, lower-risk outcome: keep the runner behavior, move it into RESTRunner.Domain, and remove RESTRunner.Services.HttpClient as a separate project.
- Existing statistics, template substitution behavior, supported HTTP verbs, and console/web integrations were preserved.
- The solution completed with a successful build and passing tests, so no additional migration phase was required for this quickfix.

## Decision Record

1. Initial direction: replace the custom execution project with WebSpark batch execution.
2. Evaluation result: direct replacement was not viable with the package version under review.
3. Final decision: consolidate `ExecuteRunnerService` into RESTRunner.Domain, keep `IExecuteRunner`, remove RESTRunner.Services.HttpClient from the solution, and preserve behavior.
4. Outcome: project structure simplified with no reported breaking changes and no regression in build or test status.
