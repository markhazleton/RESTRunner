## 1. CI Workflow — GitHub Actions for .NET 10

**Decision**: Use GitHub Actions with `actions/setup-dotnet@v4` targeting .NET 10.

**Rationale**:
- Repository already uses GitHub (`.github/` directory exists).
- `dotnet build` and `dotnet test` are the exact commands cited in the constitution Quality Gates.
- A simple push/pull_request trigger satisfies the constitution requirement without additional tooling.

**Key details**:
- Workflow file: `.github/workflows/build.yml`
- Trigger on `push` and `pull_request` to `main`
- Steps: checkout → setup dotnet `10.x` → `dotnet build --no-restore` → `dotnet test --no-build`
- Single `ubuntu-latest` runner is sufficient for cross-platform validation.
- The `dotnet restore` step runs before build to cache packages.

**Alternatives considered**:
- Azure Pipelines: rejected — no Azure DevOps org in repository.
- Multi-OS matrix: deferred — adds complexity; add once single-runner passes.

## 2. New Test Projects — RESTRunner.Web.Tests and RESTRunner.Services.Tests

**Decision**: Create two new MSTest projects mirroring the `RESTRunner.Domain.Tests` csproj pattern.

**Rationale**:
- The constitution MUST rule: "Domain and regression-prone changes MUST add or update automated tests."
- The existing `RESTRunner.Domain.Tests` defines the exact MSTest + coverlet pattern to follow.
- Splitting web tests and service tests keeps each project focused and avoids a monolithic test assembly.

**Test project structure (mirrored from Domain.Tests)**:
```
RESTRunner.Web.Tests/
├── RESTRunner.Web.Tests.csproj    (net10.0, Nullable, ImplicitUsings, MSTest 4.1.0, coverlet)
├── GlobalUsings.cs
├── Controllers/                   (Controller action tests using TestServer or direct instantiation)
└── Services/                      (Service unit tests with mocked dependencies)

RESTRunner.Services.Tests/
├── RESTRunner.Services.Tests.csproj
├── GlobalUsings.cs
└── ExecuteRunnerServiceTests.cs   (Tests for GetResponseAsync, RunAsync)
```

**csproj pattern** (from `RESTRunner.Domain.Tests.csproj`):
- TargetFramework: `net10.0`
- Nullable: `enable`, ImplicitUsings: `enable`
- MSTest.TestAdapter 4.1.0, MSTest.TestFramework 4.1.0
- coverlet.collector 8.0.0 (PrivateAssets=all)
- ProjectReference to target project

**Initial test scope** (constitution-priority high-value coverage):
- `ExecutionController`: Index (success path, empty history), Results (found, not found)
- `ConfigurationController`: Index (success), Create (valid, invalid)
- `RealExecutionService`: StartExecutionAsync (config not found, happy path)
- `ExecuteRunnerService`: GetResponseAsync (path substitution, body template substitution)

**Alternatives considered**:
- xUnit: rejected — constitution mandates MSTest.
- Single combined test project: rejected — constitution asks for per-layer coverage; separate projects enforce that boundary.

## 3. Sample Password Constants

**Decision**: Add `RESTRunner.Domain/Constants/DomainConstants.cs` with a `SamplePassword` constant.

**Rationale**:
- The `Constants/` folder does not yet exist in `RESTRunner.Domain` but is already listed in the project template tree (plan template shows `RESTRunner.Domain/Constants/`), so adding it is aligned with the intended structure.
- A single named constant makes the "sample" intent explicit, eliminates the magic string `"password"` / `"admin123"`, and prevents false positives from secret scanning tools.
- `"admin123"` in `Program.cs` is a different value from the others — having a constant prevents accidental divergence.

**Constant name**: `DomainConstants.SamplePassword = "[sample]"` — clearly not a real credential.

**Files to update** (5 production-code occurrences):
1. `RESTRunner/Extensions/CompareRunner_Extensions.cs:30`
2. `RESTRunner.Web/Program.cs:144`
3. `RESTRunner.Web/Program.cs:424`
4. `RESTRunner.Web/Program.cs:435` (currently `"admin123"` — also replaced with SamplePassword)
5. `RESTRunner.Web/Controllers/ConfigurationController.cs:452`

**Test occurrences** (SEC6, SEC7 — acceptable in test context):
- `RESTRunner.Domain.Tests/Extensions/CompareUser_ExtensionsTests.cs:19,46` — define a local `TestPassword` constant in the test class; no cross-project dependency needed.

**Alternatives considered**:
- Leave as-is with a code-level comment: rejected — comment suppression still triggers secret scanners and establishes a bad pattern.
- Use `appsettings.json` for sample values: rejected — overkill for seed data that is intentionally placeholder.

## 4. File Decomposition Strategy

**SampleCRUDClient.cs (1,259 lines)**
**Decision**: NO ACTION. File is NSwag auto-generated (confirmed by `// <auto-generated>` header and `[System.CodeDom.Compiler.GeneratedCode]` attributes). The constitution explicitly carves this out: "Auto-generated files MAY remain large when regeneration is the source of truth."

---

**FileOpenApiService.cs (632 lines)**
**Decision**: Extract `OpenApiSpecParser` internal class from the parsing/extraction region into a separate file `RESTRunner.Web/Services/OpenApiSpecParser.cs`.

**Rationale**: The file has two clearly distinct responsibilities — CRUD storage operations (IOpenApiService implementation) and OpenAPI document parsing/schema extraction. Separating them makes each unit independently testable and reduces the service to ~300 lines.

**Split boundary**:
- `FileOpenApiService.cs`: CRUD (GetAll, GetById, Save, Delete) + `IOpenApiService` interface conformance
- `OpenApiSpecParser.cs`: `ParseSpecContent()`, `ExtractEndpoints()`, `ExtractSchemaType()`, all JSON traversal helpers

---

**ExecutionController.cs (624 lines)**
**Decision**: Split into `ExecutionController.cs` (MVC view actions only) and `ExecutionApiController.cs` (minimal API / JSON endpoints only).

**Rationale**: The file already uses `// ============ MVC VIEW ACTIONS ============` and `// ============ API ENDPOINTS ============` comment headers, making the split boundary unambiguous. Both share the same injected services so constructor duplication is minimal.

**New file**: `RESTRunner.Web/Controllers/ExecutionApiController.cs` — derive from `ControllerBase`, move all `[HttpGet("/api/...")]` / `[HttpPost("/api/...")]` actions.

---

**RealExecutionService.cs (603 lines)**
**Decision**: Extract `ExecutionHistoryService.cs` for file-persistence operations, leaving `RealExecutionService.cs` as pure execution orchestration.

**Rationale**: The file has two logical concerns — running test executions (StartExecution, CancelExecution, GetStatus) and persisting/loading history. Separate `IExecutionHistoryService` interface and `ExecutionHistoryService` implementation reduces each file to well under 500 lines.

**New interface**: `IExecutionHistoryService` in `RESTRunner.Web/Services/`

## 5. BUG Comment Resolution

**Program.cs:335 — missing endpoint to list all configurations via API**
**Decision**: Implement the missing `GET /api/configurations` minimal API endpoint in Program.cs or move to `ConfigurationApiController.cs` (separate from the MVC controller).

**Rationale**: The BUG comment signals a gap in the API surface. Since `ConfigurationController` already handles MVC views, a lightweight minimal API registration or a new `ConfigurationApiController` following the same split pattern as ExecutionApiController is the clean path.

---

**ConfigurationController.cs:282 — DebugEdit action**
**Decision**: Add `[Authorize]` attribute to `DebugEdit` and restrict to non-production environments OR remove the action and track any debug need in a GitHub Issue.

**Rationale**: `DebugEdit` serializes full configuration including user Password fields (even if sample data) without any authorization gate. Simplest safe fix is adding `[Authorize]` and confirming the action is not indexed by default routes.
